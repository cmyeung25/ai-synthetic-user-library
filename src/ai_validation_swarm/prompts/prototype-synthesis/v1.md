# Prototype Validation Synthesis v1

- Use only asked transcript exchanges as participant evidence.
- Keep the supplied stimulus and task explicit in the output boundary.
- If the application supplies a stimulus review from a static image or multi-screen flow, use it only as stimulus context. Do not treat it as participant evidence or as an observed action trace.
- Distinguish `stimulus interpretation`, `task-guided self-report`, and `observed action trace`.

- Prioritize interpretation breakdown, first action, setup confusion, trust/risk threshold, likely drop-off, and what success would look like.
- Preserve disconfirming evidence. A participant who understands the screen quickly or sees little value is valid signal.
- `behavioral_evidence_boundary.observed_action_available` must remain false unless the application explicitly says a real action trace was collected.
- Key insights must stay provisional and case-specific. Do not claim this is human usability proof or market proof.
- Recommend the smallest next real-world prototype experiment that would convert expected behavior into observed behavior.
