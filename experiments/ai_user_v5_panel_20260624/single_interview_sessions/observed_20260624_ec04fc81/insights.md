## Current Workaround

- Pain: medium; switching: medium
- 先睇 funnel 同 drop-off。
- 翻 support ticket 同 call note，對卡位是否一致。
- 有 replay 或 prototype 就用嚟分辨係理解問題定 flow 太重。
- 如果新訊號同現有解讀唔同，會返去對 event、replay、support case 先。

## Trust Boundary

- 要分清楚邊啲係啟發，邊啲唔可以當 evidence。
- 要有 evidence trace，而唔只係一段似真說話。
- 要明確標示只適合早期假設同問題 framing。

## First Value

- 幫佢喺未排到真人訪談之前，快啲見到有冇明顯盲點。
- 提出一個值得驗證嘅新假設，而呢個假設可以再用 event、replay、support case 去交叉驗證。
- 幫到下一輪驗證方向調整，而唔係直接叫佢改 roadmap。

## Key Insights

- 呢位 PM 做 roadmap / feature priority 時，核心唔係搵一個『正確答案』，而係喺證據未完備下，搵到一個公開可 defend、私下仍可保留、而且成本同時機合理嘅試法。
- 佢而家已經有一套輕量 triage workflow：analytics、support、call note、replay、prototype 交叉比對。新工具如果唔能夠縮短呢步，反而要佢再做一次驗證，就會被視為負擔。
- AI synthetic-user platform 對佢最合理嘅位置係 early framing，不係決策終局器，更唔係真人研究替代品。
- 信任門檻非常具體：要講得出假設、input、可能出錯位，仲要同現有 evidence 對得返一部分。
- 最強 adoption signal 係『幫我產生一個值得驗證、而家未諗到嘅新假設』；最強 rejection signal 係『講得似真，但 generic，同真數據對唔上』。

## Next Experiment

用真實 onboarding case 做一個最細 workflow test：輸入現有 analytics、support feedback 同 prototype context，產出 3-5 個 synthetic objection / confusion hypotheses，每個都附假設、input basis、可能錯位。然後觀察 PM 會唔會用其中任何一個去改下一輪驗證方向，並量度佢為核實輸出額外花咗幾多時間。
