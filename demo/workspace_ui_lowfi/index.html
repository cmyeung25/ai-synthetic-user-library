The main canvas starts from what the team wants to learn, not from mode pickers or a large setup form.

Live research plan

This panel updates as the conversation becomes more structured. It is the bridge between conversational intake and the real run contract.

What the system knows

Research intent: trust and adoption risk for an AI operations assistant. Available materials: founder brief and onboarding screenshots.

What remains implicit

Mode, panel shape, and evidence class are still hidden until enough context is gathered.

Inferred setup

Prototype validation is the likely primary mode because screenshots exist. Adoption risk remains a secondary synthesis lens.

Minimal follow-up

Ask only for missing market slice and panel stance, not full config schema.

Ready to submit

The system now exposes the exact inferred plan so the user can confirm, correct, or open advanced controls.

Advanced path

Optional mode override, persona filters, quota-aware sampling, and provider selection sit behind a secondary panel.

After submit

The workspace shifts to queue visibility, evidence retrieval, and replay actions. This is still a research console, not a generic analytics dashboard.

Why this matters

The user can inspect why the plan ran, what evidence was produced, and where human validation is still required.

Execution Summary

Plan sidecar

The right rail is always explicit about evidence boundaries and operational constraints, even though the intake path stays low-friction.

Evidence boundary

Do not imply observed behavior until the system confirms the stimulus supports it. At intake time this remains unresolved.

Complexity guardrail

The default path should not expose mode selectors, workflow graphs, or raw panel schema fields.

Operational guardrail

Quota, role, billing status, and retention remain visible before execution so the conversational path does not hide real runtime constraints.
