Layout rules

The default shell is a quiet operations console. It should feel like a place to run studies and inspect evidence, not to admire summary cards.

  • Use a left rail for workspace location, not for decorative navigation chrome.
  • Keep the main canvas focused on conversational intake and run review.
  • Use the right rail for explicit plan, evidence boundary, and runtime constraints.
  • Keep dense information readable with low-radius panels, restrained motion, and strong borders.
  • Reserve bright accent color for action, inferred-plan emphasis, and bounded state changes.

Named next actions

The main button should describe what happens next rather than hiding behind a generic disabled submit control.

The remediation contract should tell the UI whether to ask a question, request an artifact, confirm a fallback, or resume later.

Implementation use

This theme file is intended to be reused by future HTML prototypes and any eventual workspace frontend shell before a framework-specific component library exists.

  • Use `workspace-theme.css` as the shared token and component layer for workspace-console prototypes.
  • Keep new pages on the same surface language unless there is a product reason to break the pattern.
  • Add advanced controls as extensions of this system, not as a separate visual language.
