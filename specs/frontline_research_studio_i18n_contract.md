# Frontline Research Studio I18n Contract

Status: implemented M36 contract slice.

Owner: platform-development-chief + platform-system-architect.

Last updated: 2026-07-02.

Companion terminology contract: `specs/frontline_research_studio_bilingual_terminology_glossary.md`.

## Alignment Check

- Research bottleneck improved: ordinary users need to set up, run, review, decide, and share studies without English-only product chrome becoming a workflow barrier.
- Primary improvements: scalable research throughput, evidence discipline, and user-facing workflow usability.
- North-star fit: multi-language chrome helps the platform replace parts of interviewer-led setup and synthesis work for more users, but only if translation does not rewrite evidence provenance or overstate synthetic signal.

## Supported Locales

The first supported locales are:

- `en`: deterministic default product chrome.
- `zh-Hant`: explicit Traditional Chinese product chrome.

Locale selection precedence:

1. URL query parameter `?lang=...`.
2. Persisted browser preference in `localStorage.frontline_studio_locale`.
3. Default locale `en`.

The browser document `lang` attribute must follow the resolved locale.

## Product Scope

The i18n layer owns product chrome only:

- route labels and route headers
- left-rail navigation and account language switcher
- workspace, project, study setup, Research Copilot, Persona Library, plan confirmation, and plan tuning copy
- run, evidence, saved evidence view, report, decision, and share route chrome
- buttons, empty states, loading states, metric labels, action notices, and user-facing recovery text
- synthetic-evidence boundary copy that belongs to the product surface

For `zh-Hant`, product chrome must follow `specs/frontline_research_studio_bilingual_terminology_glossary.md`. Stable user-facing terms should use the approved Traditional Chinese labels, such as `專案`, `研究`, `研究計劃`, `研究執行`, `證據`, `決策`, and `合成受訪者`.

The i18n layer does not own generated research content:

- transcripts
- source evidence summaries
- findings
- report body content generated by the runtime
- decision summaries authored by the user or runtime
- persona artifacts, profile biographies, and provenance records
- backend JSON artifacts, audit records, run manifests, and calibration outputs

Generated research content must stay in its original language unless a future evidence-translation contract records translation provenance, original language, translated language, model/provider lineage, and review status.

## Evidence Boundary

Translation must not create a new evidence claim.

Rules:

- Product chrome may be localized to improve comprehension.
- Evidence, transcript, finding, and report-body content must not be silently translated in the UI.
- If evidence translation is later introduced, the translated text must be marked as a derivative view, not a replacement source.
- Shares and exports must preserve the synthetic-evidence boundary in the selected product locale without modifying underlying artifacts.

## Architecture Boundary

Frontend implementation:

- `frontend/frontline_research_studio/src/i18n.js` owns supported locales, default locale, locale normalization, initial locale resolution, `html lang` mapping, and dictionary lookup.
- `frontend/frontline_research_studio/src/main.jsx` consumes i18n through one React context and passes translated labels into route-level product chrome.
- Locale persistence is a browser convenience and must not be required for backend correctness.
- `zh-Hant` dictionary entries must use the bilingual terminology glossary for durable product objects, workflow states, synthetic participant labels, and evidence-boundary copy.

Backend implementation:

- Existing backend APIs remain locale-neutral for the M36 i18n slice.
- API payloads continue returning canonical object fields and evidence artifacts in their original language.
- The backend must not infer that `zh-Hant` UI usage changes study language, participant market, persona locale, calibration market, or evidence validity.

Future SaaS/cloud implementation, if evidence translation is added:

- Add explicit fields such as `content_language`, `original_language`, `translated_language`, `translation_status`, `translation_provider`, `translation_model`, `translation_source_hash`, and `translation_review_state`.
- Store translations as derivative artifacts or view projections.
- Keep original artifacts immutable and cite them as the source of truth.

## Routing and IA Rules

I18n must preserve the route-aware product model:

- `/studio`
- `/studio/projects`
- `/studio/projects/{project_id}`
- `/studio/studies/new`
- `/studio/studies/{study_id}`
- `/studio/studies/{study_id}/setup`
- `/studio/studies/{study_id}/runs`
- `/studio/studies/{study_id}/runs/{run_id}`
- `/studio/studies/{study_id}/evidence`
- `/studio/studies/{study_id}/evidence-views/{evidence_view_id}`
- `/studio/studies/{study_id}/reports/{study_report_id}`
- `/studio/studies/{study_id}/decisions/{decision_log_id}`
- `/studio/share/{share_bundle_id}`

Browser refresh, back/forward, and deep links must preserve the selected durable object and evidence boundary independent of language selection.
